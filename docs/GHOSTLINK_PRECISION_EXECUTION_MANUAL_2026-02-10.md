# GhostLink 정밀 실행 매뉴얼 (2026-02-10)

## 목적

제품 기능을 "추정 중심"이 아닌 "증거 중심"으로 운영하기 위한 실행 기준을 정리합니다.

## 현재 기준선

- 결제/승인/최적화 루프가 서비스 계층으로 분리되어 있음
- 프루프 센터(Proof Center) 지표 계산 로직이 존재함
- 일일 리포트 생성 파이프라인이 동작함
- 관리자 CRM 화면으로 운영 대응 속도를 높일 수 있음

## 정밀 실행 원칙

1. 증거 우선
- 모든 개선 주장은 측정값 또는 예측값 라벨을 가져야 함

2. 권한 우선
- 민감 작업(결제 변경, 승인 처리)은 역할 기반 제어

3. 실패 허용
- 감사 로그, 외부 이벤트 실패가 핵심 비즈니스 요청을 중단시키지 않도록 설계

4. 재현 가능
- 같은 입력에 같은 결과를 만들 수 있는 테스트/로그 체계 유지

## 운영 루프

1. 데이터 수집
- Bridge 이벤트, Proof 런 결과, 구독/결제 이벤트 수집

2. 판단
- v1/v2 최적화 루프에서 후보 생성 및 우선순위 결정

3. 승인/실행
- owner/admin 승인 후 실행

4. 측정
- 전후 지표 비교, 보상/피드백 반영

5. 보고
- 대시보드/리포트/관리자 화면에 일관 노출

## 실무 체크리스트

- 릴리스 전
  - 결제 플로우 스모크 테스트
  - 웹훅 멱등성 확인
  - 승인 플로우 권한 확인
- 릴리스 후
  - 24시간 500 에러 모니터링
  - 감사 로그 이상치 확인
  - 인보이스 적재 실패 여부 확인

## 팀 역할

- Product: 지표 해석 기준과 고객 커뮤니케이션 정의
- Backend: 상태 전이/멱등성/오류 격리 책임
- Frontend: 지표 라벨/상태 노출 일관성 책임
- CS/Ops: 고객 대응, 이슈 분류, 재현 정보 축적
